Using HArvest

WordPress

Install the HArvest WordPress plugin, configure it with your HA URL, then use shortcodes to embed widgets on any page or post.

Requirements

RequirementVersion
WordPress5.0+
PHP7.4+ (8.0+ recommended)
EditorAny context that processes shortcodes for [harvest], or any raw HTML/code context for the hrv-mount div format. Classic Editor is optional.

The plugin works with any WordPress theme. It enqueues the widget script in the page footer and adds the configured Home Assistant origins to the page's Content Security Policy header. If another plugin or your web server manages CSP separately, you may need to add the origins there instead.

Installation

  1. 1

    Download the plugin

    Download the latest release zip from GitHub Releases. Inside the zip, find harvest.zip in the wordpress/ folder. This is the WordPress plugin.

  2. 2

    Upload to WordPress

    In your WordPress admin, go to Plugins > Add New > Upload Plugin. Select the ZIP file and click Install Now, then Activate Plugin.

  3. 3

    Configure the plugin

    Go to Settings > HArvest and enter your Home Assistant external URL. Save the settings.

Plugin settings

The settings page is at Settings > HArvest in your WordPress admin.

SettingDescription
Home Assistant URL Required. The public widget transport origin reachable by your WordPress visitors, for example https://myhome.duckdns.org. To use HArvest alternate-port transport, enter the alternate origin including its port, for example https://ha.example.com:9050. Public and HTTPS-hosted pages must use an HTTPS origin. The plugin uses this value for shortcode WebSocket connections, HA-served widget assets, preconnect, and Content-Security-Policy directives.
Widget JS source Two script-source options, plus alternate-port transport configured through the Home Assistant URL:
  • HA-served (recommended, default) - the widget JS is fetched from your Home Assistant instance at {your-HA-URL}/harvest_assets/harvest.min.js. Updating HArvest via HACS automatically updates the widget on every WordPress page. Requires the Home Assistant URL field above to be set (the radio is disabled otherwise).
  • Alternate-port transport - keep HA-served selected and enter the alternate origin, including its port, in the Home Assistant URL field. The alternate server carries the widget script, WebSocket connection, and runtime assets. It is plain HTTP unless placed behind a TLS proxy or secure tunnel.
  • Custom URL - point only the widget script at a trusted self-hosted copy of harvest.min.js. The live WebSocket transport still uses the Home Assistant URL field. Accepts full URLs (https://example.com/harvest.min.js), absolute paths (/harvest.min.js), and relative paths or bare filenames (harvest.min.js). Relative values resolve against the visitor's current WordPress page URL.

Live URL reachability check

As you type or change either URL field, the plugin probes public HTTP and HTTPS URLs from the WordPress server and shows a small status indicator next to the field. For security, the check refuses to request private, loopback, link-local, or other internal network addresses. Every redirect destination is checked under the same rules.

  • ✓ (green) - the URL responded with HTTP 2xx. Reachable.
  • ⚠ (amber) - the URL did not respond, or returned a 4xx/5xx. Doesn't necessarily mean the widget will fail for visitors: your visitors may be on a different network than your WordPress server (e.g. a LAN-hosted Home Assistant). Save anyway if you know the URL is correct from a visitor's perspective.
  • ⓘ (amber) - the URL is a relative path or bare filename. Cannot be probed from the server side because it only resolves against the visitor's embed page; you'll need to verify it manually.
  • ✗ (red) - the URL is malformed, uses a disallowed scheme (javascript:, data:, etc.), or resolves to an internal network address. Malformed URLs and disallowed schemes cannot be used. Internal URLs may still be saved, but the reachability check will not request them.

The check is informational only and never blocks save. Verify internal URLs from the network where widget visitors will use them.

Shortcodes

[harvest] - single card

Embeds a single widget card for one entity.

[harvest token="hwt_..." entity="light.bedroom_main"]
ParameterRequiredDescription
tokenYesWidget token ID from the HArvest panel
entityYes*HA entity ID. e.g. light.bedroom_main
aliasYes*8-character alias instead of entity ID. Use entity OR alias, not both.
token-secretNoHMAC signing secret, required when the token has HMAC authentication enabled
Tip

The HArvest panel's Code section generates the exact shortcode for each entity - you don't need to type these manually. Just copy and paste.

[harvest_group] - multiple cards

Groups multiple [harvest] shortcodes together, sharing a token and HA URL. Equivalent to wrapping cards in an <hrv-group> element.

[harvest_group token="hwt_..."]
  [harvest entity="light.living_room"]
  [harvest entity="light.dining_room"]
  [harvest entity="switch.lamp"]
[/harvest_group]
ParameterRequiredDescription
tokenYesToken ID shared by all cards in the group
token-secretNoHMAC secret shared by all cards in the group

When using [harvest_group], child [harvest] shortcodes do not need token - they inherit it from the group.

Using aliases

Use the alias format when you do not want real Home Assistant entity IDs visible in page source. Aliases reduce information disclosure but do not change the token's permissions or provide an additional security boundary. The panel's Code section shows both formats.

[harvest token="hwt_..." alias="dJ5x3Apd"]

Rendered output

The shortcodes render as a <div class="hrv-mount"> element with the appropriate data-* attributes. The widget script mounts an <hrv-card> custom element inside each mount div at runtime. The script is enqueued in the page footer (once per page, regardless of how many shortcodes are on the page).

Content Security Policy

If a page sends a Content Security Policy, its connect-src directive must allow the WebSocket origin and its script-src directive must allow runtime scripts from Home Assistant. The plugin creates or updates these directives based on the HA URL you entered in the plugin settings.

The injected header looks like:

Content-Security-Policy: connect-src 'self' wss://myhome.duckdns.org; script-src 'self' https://myhome.duckdns.org

The plugin maps the URL scheme from settings to its WebSocket equivalent: https:// becomes wss://, and http:// becomes ws://. It also permits the configured HTTP or HTTPS origin in script-src so renderer scripts can load.

Existing CSP headers

If your WordPress host or another plugin sets CSP through a different mechanism, the HArvest plugin's changes may be ignored or replaced. In that case, add the matching WebSocket URL to connect-src and the Home Assistant HTTP or HTTPS origin to script-src in the policy managed by that system.

Script loading

The widget script (harvest.min.js) is loaded in the page footer via WordPress's wp_footer action. It's only enqueued once per page even if multiple shortcodes are present.

The script handle is harvest-widget. If you need to reference it from another plugin or theme (e.g. to add a dependency), use this handle.
