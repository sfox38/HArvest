Dashboard

The dashboard is the first screen you see when you open the panel. It shows a real-time snapshot of what's happening across all your widgets.

Stat cards

Five sensor cards across the top of the dashboard:

CardWhat it shows
Active SessionsNumber of widget sessions currently connected
Active TokensTotal non-expired, non-revoked tokens
Commands TodayService calls sent through HArvest in the last 24 hours
Errors TodayAuth failures and flood protection events in the last 24 hours
DB SizeCurrent activity log database size

Activity graphs

Three time-series graphs covering the last 24 hours: commands per hour, active sessions over time, and auth failures per hour. These update in real time as events arrive.

Sidebar sections

Below the graphs: a quick view of the last four activity events (click any to open the widget detail), and a list of widgets expiring in the next 7 days.

Version drift banner

When one of your sites is running a HArvest WordPress plugin or widget bundle that's older than your HA integration, a yellow banner appears at the top of the Dashboard listing each affected site. This typically happens when you've updated HArvest in HACS but haven't updated the WordPress plugin yet, or when a visitor's browser is loading an old cached version of the widget JS. The banner tells you exactly what's outdated and how to fix it.

Click Dismiss to hide the banner. It stays hidden until your next HArvest update, at which point any drift that still exists reappears for re-evaluation. There is no per-site dismiss; one click hides everything from this version onward.

Widget list

The Widgets tab lists all your widgets. Each row shows the widget name, status badge, primary origin (or "any website"), entity count, active session count, and expiry date.

Status badges

StatusMeaning
ActiveWidget is valid and accepting connections
PausedManually paused - existing sessions are dropped, new connections rejected
Expiring soonExpires within 7 days
InactiveHas an active schedule set, currently outside the allowed hours
ExpiredPast its expiry date
RevokedPermanently revoked - all sessions dropped, widget cannot be re-enabled

Filtering and search

Filter by status with the tabs at the top (All, Active, Expiring, Paused). The search box does multi-word fuzzy matching on widget names and origins. Archived (revoked/expired) widgets are collapsed into a separate section at the bottom of the list.

Widget detail

Clicking a widget opens the widget detail screen. This is where you manage everything about a specific widget: its entities, access rules, appearance, and embed code.

The header shows the widget name (editable inline), a status badge, and action buttons:

  • Pause / Resume - temporarily suspend the widget, dropping all active sessions. Resume restores normal operation.
  • Duplicate - create a new widget that copies all settings (entities, origins, theme, display, rate limits) from this one. The copy gets a fresh token, fresh aliases, no expiry, and no HMAC secret. The name is set to "... (copy)".
  • Revoke - permanently disable the widget. This cannot be undone.

The detail screen uses a single-column layout with five tabs across the top:

  • Entities - entity management, theme selection, live preview, and per-entity settings
  • Embed - code snippets for HTML and WordPress
  • Preferences - origins, schedule, display defaults, and rate limits
  • Security - token secret, IP allowlists, and session lifetime limits
  • Widget Info - usage stats, active sessions, and recent activity

Entities tab

The Entities tab is a split panel. The left side is an entity list; the right side shows the theme picker, a live preview, and per-entity settings for whichever entity you have selected.

Entity list (left panel)

Each row shows a domain icon, the entity ID, and an alias badge (if set). Click a row to select it. Only primary entities appear here; companion entities are managed inside the selected entity's settings card. Use the search input at the top to add new entities with fuzzy matching.

Theme card

The theme picker is always visible at the top of the right panel, even when no entity is selected. It shows a horizontal scrollable strip of theme thumbnails. A text filter below the strip narrows the list by theme name or ID. Themes that include a renderer override are marked with a ☆ badge. Selecting a theme applies it to the widget and updates the live preview immediately.

Live preview

When an entity is selected, a live preview card appears below the theme picker. It renders the selected entity using the current theme and all applied settings. The preview updates automatically as you change settings, switch themes, or modify display options. It also respects the entity's forced color scheme setting (Auto, Light, or Dark).

Entity settings card

Below the preview, the settings card shows all configuration for the selected entity. Settings are grouped into universal controls (available for all entity types) and domain-specific controls.

This section auto-opens when companions already exist. Hidden when Access is Badge. Each companion has a small "Read only" toggle (default: on). "Perform action" calls a Home Assistant service (e.g. light.turn_on) with optional service data in JSON format. The service must be in the entity domain's allowed service list.

When Access or Always use is set to "Per entity", selecting an individual entity in the list shows that entity's own Access or color scheme toggle. Each entity can also have its own decimal rounding setting for numeric domains.

Embed tab

Ready-to-paste embed snippets that update in real time as you change entity or alias settings. The snippet panel has two tabs (HTML and WordPress) and a toggle to switch between entity= and alias= format. The alias format is useful when you don't want entity IDs visible in your page source.

Preferences tab

Collapsible sections for:

  • Origins - which websites can use this widget. Add specific URLs (e.g. https://mysite.com) with optional path restrictions, or enable "allow any origin".
  • Schedule - restrict access to specific days of the week and hours of the day, in a chosen timezone.
  • Rate limits - per-widget override for the global rate limit defaults.
  • Display defaults - language, accessibility mode, and offline/error message settings.

Security tab

Token secret for HMAC-signed auth, IP allowlists, and session lifetime limits. See the Security page for details on each option.

Widget Info tab

Usage stats (total sessions, commands, last connection time), a live list of active sessions with the ability to end individual sessions, and a recent activity feed for this widget.

Widget wizard

The wizard guides you through creating a new widget in four steps: Design, Origin, Expiry, and Done. It's accessible from the + Create Widget button on the dashboard or widget list.

The wizard is covered step-by-step in the Quick setup page. Additional details on the options available at each step:

Single vs. group mode

Step 1 (Design) has a toggle between Single card, Group of cards, and Page of cards modes. Switching between modes preserves your selected entities. Single mode keeps only the first entity. In Group and Page modes, clicking an entity row selects it for preview, and a [+] badge on each row lets you add companion entities inline.

Live preview

Step 1 shows a live preview of your selected entity as you configure it. The preview updates in real time when you change the permission level, select a different entity, or pick a theme. Companions also appear in the preview.

Script variables

When you add a script entity with View and control permission, the wizard shows a form for the script's input variables (if any are defined in the script's YAML fields: config). Each variable renders as a typed input matching its HA selector: dropdowns for areas, floors, devices, entities, and select lists; number inputs with min/max constraints; checkboxes for booleans; and text inputs for everything else. Default values from the script definition are pre-filled. These values are baked into the widget and sent automatically when a visitor clicks Run. The same form is available in the widget detail editor under Script variables.

Alias toggle (Step 4)

The code snippet in Step 4 (Done) has an Aliases toggle. When enabled, the snippet uses 8-character random aliases instead of real entity IDs in the embed code. Both formats work - aliases are just privacy aliases, not a different access mechanism. Aliases for each entity are generated when you select the entity in Step 1 and are stored with the widget permanently.

Embed modes

The snippet panel shows two embed modes:

  • Custom element - uses <hrv-card> and <hrv-group> custom HTML elements
  • Data attribute - uses plain <div class="hrv-mount"> elements with data- attributes, for environments where custom elements are blocked or inconvenient

Dashboard converter

The dashboard converter lets you turn an existing HA Lovelace dashboard into a standalone HTML page with HArvest widgets. Open it from the Convert button in the Widget list toolbar.

How it works

  1. Select a dashboard - pick from your installed Lovelace dashboards (default or custom).
  2. Configure - choose which views to include, set entity capabilities (Smart, Read only, Control, or Badge), pick a theme, and optionally restrict the origin.
  3. Generate - the wizard creates HArvest widgets (one per view, automatically split if a view exceeds 50 entities) and renders the HTML.
  4. Download - save the generated HTML file. Upload it to your web server, open it locally, or paste the code into your site.

What gets converted

The converter handles most standard Lovelace card types: single-entity cards (light, thermostat, sensor, tile, etc.), multi-entity cards (entities, glance), stacks and grids, heading cards with inline badges, and view-level badges. Text-only cards appear as styled text blocks. Unsupported cards show a placeholder with contextual information.

Custom card support

Popular custom card sets have dedicated conversion support:

  • Mushroom cards - all single-entity mushroom cards (entity, light, fan, cover, climate, lock, media-player, vacuum, humidifier, number, update, alarm, person, select) are converted with proper entity extraction. Mushroom chips cards convert to HArvest badges. Template cards are skipped (Jinja templates cannot be converted).
  • Bubble Card - button, climate, cover, media-player, select, and calendar card types are converted. Sub-buttons are automatically mapped to HArvest companion entities so they appear alongside the primary card. Separators, empty columns, and pop-ups are skipped.
  • Button Card - the primary entity is extracted and converted. Template-based custom fields and variables that reference entities via JS templates are not converted.
  • mini-graph-card - all entities from the card's entities list are extracted and converted.
  • mini-media-player - the primary entity is extracted and converted.
  • simple-thermostat - the primary entity is extracted and converted.
  • Stack containers - stack-in-card, vertical-stack-in-card, layout-card, swipe-card, and tabbed-card are treated as layout containers and their child cards are recursively converted.

Other custom cards not listed above still have their entities extracted via a deep scan of the card config, though this may pick up entity IDs from action targets or template variables that are not the card's primary entity.

Limitations

  • Tier 3 entities (cameras, alarm panels, device trackers, scenes, updates) are excluded for security.
  • Visual styling is not copied from Lovelace; cards use the selected HArvest theme instead.
  • Bubble Card sub-buttons are auto-detected as companions, but other companion relationships are not. You can add companions manually after conversion.
  • Gesture actions that HArvest supports (toggle, none, call-service / perform-action) are converted. Other Lovelace actions (more-info, navigate, url) have no HArvest equivalent and are skipped.
  • Conditional card logic is dropped (the inner card is included unconditionally).

Created widgets are labeled with a "Converted:" prefix so they are easy to identify in the widget list. You can edit any converted widget's settings, entities, theme, or security options after conversion.

Themes

Theme selection is per-widget, managed in each widget's Entities tab. Each widget picks its own theme, and the selection affects all entities within that widget. Creating and editing themes happens in the dedicated Themes tab.

HArvest ships five bundled themes (Default, Glass, Access, Minimus, Shrooms) and supports unlimited custom themes and user-contributed themes.

Theme picker strip

The theme card at the top of the Entities tab's right panel shows a horizontal scrollable strip of theme thumbnails. Bundled themes are marked with a System badge. Three indicators can appear overlaid on a theme thumbnail: a ★ badge when the theme includes a renderer override (shown as ⚠ if the renderer JS file is missing), and an F badge when the theme bundles a custom font (hovering shows the font family names). A text filter below the strip narrows results by theme name or ID.

Renderer overrides

Some themes include a renderer override, which replaces the default card renderers with custom designs. When a theme with a renderer is active, the domain-specific controls in the entity settings card are filtered to show only options the renderer supports. Switching between themes preserves your display settings; options the new theme does not support are hidden but kept in storage.

Custom themes

Custom themes are created and managed in the Themes tab. The theme JSON editor lets you modify CSS variables and preview changes in real time. Themes are exported as .zip files. See the Theming page for the full theme JSON format and variable reference.

Sharing themes

Any theme can be exported as a .zip file. Custom themes can be imported on any HArvest instance. This is how third-party themes are distributed: the theme author exports the zip and shares it, and other users import it via the Themes tab.

Settings

The Settings screen has collapsible sections for every global configuration option.

Allowed origins

A registry of websites that widgets can reference. Rather than typing a URL each time you create a widget, you add sites here once with a friendly name. These then appear as options in the wizard and widget detail screen. You can add origins directly from the widget editor too - they sync to this registry.

Both URL fields show a live reachability indicator that probes the URL from your Home Assistant server (so it works regardless of browser-side CORS):

  • ✓ (green) - the URL responded with HTTP 2xx. Reachable.
  • ⚠ (amber) - the URL did not respond, or returned a 4xx/5xx. Doesn't necessarily mean visitors will fail to load the widget: visitors may be on a different network than HA. Save anyway if you know the URL is correct.
  • ⓘ (amber) - the URL is a relative path or bare filename. Cannot be probed because it only resolves against the visitor's embed page; you'll need to verify it manually.
  • ✗ (red) - the URL is malformed or uses a disallowed scheme. The integration will refuse to use it.

The check is informational only and never blocks save.

Sensitive domains

Six Tier 1 domains are disabled by default because their actions carry higher real-world risk. Five settings control them: the Button setting governs both button and input_button. When disabled, entities from the affected domain are hidden from the entity picker, rejected at token creation, and silently dropped from active sessions.

DomainDefaultRisk if enabled
lockDisabledPhysical lock/unlock control
coverDisabledGarage doors, gates, motorized blinds
scriptDisabledArbitrary HA script execution
automationDisabledTrigger, enable, or disable HA automations
button, input_buttonDisabledOne-shot press actions, controlled together by the Button setting

Custom domains

The Custom domains card extends HArvest's server-side service allowlist for Tier 2 domains, such as third-party integration domains. It is a global setting shared by all widgets. It does not add controls to the generic card or grant Control access to an entity.

  1. Select a domain. The dropdown only lists non-built-in, non-blocked domains that currently have at least one entity and registered HA service.
  2. Select the specific services that HArvest widgets may call.
  3. Click Add. The allowlist applies immediately to current and future sessions.

To change a domain's services, remove it and add it again. Removing it immediately blocks later commands without disconnecting active sessions. Commands still require the entity to be included in the widget token with Control access, and must be sent by a configured gesture or custom renderer.

Security note

Custom Domains permits actions globally. Enable only the services your widgets require. Tier 1 domains use HArvest's built-in allowlist, and Tier 3 domains cannot be added.

Configure retention period (default 30 days). Shows current database file size and path. The activity log uses SQLite and is stored in your HA config directory.

Activity log

The Activity tab shows a searchable, filterable log of all events across all tokens. Events are retained for the configured retention period (default 30 days).

Filters

  • Date range - Last 24 hours, Last 7 days, Last 30 days, or custom
  • Widget - All widgets, or a specific widget
  • Event type - All, AUTH_OK, AUTH_FAIL, COMMAND, SESSION_END, TOKEN_REVOKED, etc.

Event rows

Each event row shows the event type with a color-coded icon, a summary, the widget name (clickable to open that widget's detail screen), and the relative timestamp. Click any row to expand it and see the full event payload: session ID, origin, entity ID, action taken, IP address, and error code where applicable.

CSV export

The Export CSV button at the top of the Activity tab exports all events matching the current filter criteria as a CSV file.
