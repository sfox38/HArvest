Three styling tiers

HArvest offers three independent levels of customization. Each is more powerful and more complex than the last - use the simplest one that solves your problem.

Tier 1 - CSS Custom Properties

The primary theming mechanism. Define color, typography, spacing, and radius values via CSS custom properties (variables). These are set server-side per token and pushed to the widget over WebSocket during authentication. No client-side configuration needed.

This covers the vast majority of use cases. Start here.

Tier 2 - ::part() selectors

Every meaningful element inside each card's Shadow DOM has a part="..." attribute. Target these from your host page's stylesheet to override structural CSS that isn't exposed as a variable - for example changing flex direction, removing borders, applying a specific font metric, or hiding elements.

/* Target the toggle button on all light cards */
hrv-card::part(toggle-button) {
  background: hotpink;
  border-radius: 0;
}

/* Target brightness slider */
hrv-card::part(brightness-slider) {
  accent-color: coral;
}

Tier 3 - Custom renderers

Replace the entire card UI for any entity domain with your own class. Full control over HTML structure, CSS, and behavior. Custom renderers are registered via HArvest.registerRenderer() or bundled as a renderer.js file inside a theme zip. See Custom renderers below.

Theme JSON format

Themes are plain JSON objects stored server-side in the HArvest integration. The panel's Themes tab is where you create and manage them. Theme variables are pushed to connected widgets over WebSocket during auth and update live when a theme is changed.

{
  "name": "Midnight Blue",
  "author": "your-name",
  "version": "1.0",
  "harvest_version": 1,
  "variables": {
    "--hrv-color-primary": "#4a90d9",
    "--hrv-color-surface": "#f8f8ff",
    "--hrv-color-text": "#1a1a2e",
    "--hrv-card-radius": "16px"
  },
  "dark_variables": {
    "--hrv-color-surface": "#1a1a2e",
    "--hrv-color-text": "#e0e0ff",
    "--hrv-color-border": "#2a2a4e"
  },
  "has_renderer": true
}
FieldTypeRequiredDescription
namestringYesDisplay name in the panel theme picker
harvest_versionintegerYesMust be 1
variablesobjectYesCSS custom property key-value pairs for light mode
authorstringNoTheme author name
versionstringNoTheme version string
descriptionstringNoShort description shown in the panel theme list.
dark_variablesobjectNoVariable overrides for dark mode. Only changed keys needed.
has_rendererbooleanNoSet automatically when the theme zip includes a renderer.js file. Read-only on import.
capabilitiesobjectNoDomain-specific feature declarations. Controls which per-domain options appear in Entity Settings.
icon_setstringNoIcon set the theme renders entity icons in: fa, tabler, ph, or a Phosphor weight (ph-thin, ph-light, ph-bold, ph-fill, ph-duotone). Empty or absent means Material Design. Icons without an equivalent stay Material Design; per-entity icon picks and the widget's own Icon set preference override the theme. Editable in the panel theme info for user themes.

Theme zip format

Import and export use a .zip file. The zip can contain:

  • theme.json (required) - the theme definition.
  • renderer.js (optional) - a custom renderer override. Replaces built-in card renderers when this theme is active.
  • *.woff or *.woff2 (optional) - custom font files. Up to 8 files. See Custom fonts below.
  • thumbnail.png or thumbnail.jpg (optional) - a preview image shown in the theme picker.

The zip file itself can be named anything (e.g. my-cool-theme.zip). Maximum upload size is 10 MB. When renderer.js is present, the panel shows a renderer consent dialog before completing every import, including overwrites. Exporting a theme produces a zip in the same format, including any thumbnail, fonts, and renderer associated with it.

HArvest validates the archive's runtime asset files before installing or overwriting the theme. Font files must use a basename ending in .woff or .woff2, and their contents must match that format. Renderer files must contain valid UTF-8 JavaScript. Invalid runtime assets are rejected without changing the installed theme.

Advanced users may copy matching theme JSON and renderer JavaScript files directly into HArvest's user theme and renderer directories. Direct filesystem copies do not pass through the import dialog. They remain disabled until the integration-wide renderer consent gate has been accepted, and theme reloads do not bypass that gate.

Unknown fields and unknown variable names are silently ignored, which makes themes forward-compatible. Only use variable names from the published list below.

Custom fonts

Themes can bundle custom fonts to give cards a distinctive typeface. This is useful when your theme's visual identity depends on a specific font that users won't have installed locally.

Bundled fonts

Bundle a WOFF2 font file inside the theme zip. The widget injects @font-face rules into the page so the font is available inside card shadow DOMs.

Step 1: Place your .woff2 file alongside theme.json in the zip:

theme.json
nunito.woff2

Variable fonts (a single file covering multiple weights) are recommended. They keep the zip small and require only one custom_fonts entry.

Step 2: Add a custom_fonts array to your theme.json:

{
  "name": "My Theme",
  "harvest_version": 1,
  "custom_fonts": [
    {
      "family": "Nunito",
      "url": "nunito.woff2",
      "weight": "200 1000",
      "style": "normal"
    }
  ],
  "variables": {
    "--hrv-font-family": "'Nunito', system-ui, sans-serif"
  }
}
FieldRequiredDescription
familyYesThe printable CSS font-family name to register, up to 100 characters
urlYesBasename of the .woff or .woff2 file in the zip
weightNonormal, bold, a numeric weight from 1 to 1000, or a numeric range such as "200 800". Defaults to "normal".
styleNonormal, italic, or oblique. Defaults to "normal".

Step 3: Reference the font in --hrv-font-family, quoted, with fallbacks:

"--hrv-font-family": "'Nunito', system-ui, sans-serif"

Always include fallback fonts. Custom fonts load asynchronously with font-display: swap - text renders immediately in the fallback and swaps once the font is ready.

Finding free fonts

Google Fonts has thousands of freely licensed fonts. To get the WOFF2 URL, request the CSS, as in this example:

curl -H "User-Agent: Mozilla/5.0" "https://fonts.googleapis.com/css2?family=Font-name:wght@400;700&display=swap"

The response will contain the direct download URLs for your font.

If your theme needs separate files for different styles (e.g. regular and italic), add multiple entries to the array. Each gets its own @font-face rule.

Updating fonts: There is no separate font upload UI. To change fonts on an installed theme, export the zip, replace or add font files, update custom_fonts in theme.json, and re-import with overwrite enabled.

Alternative: Google Fonts CSS

Instead of bundling a font file, you can load fonts from Google Fonts by adding a <link> tag to your page. The @font-face rules from Google's stylesheet are global and apply inside shadow DOMs via font-family inheritance, so no custom_fonts array is needed.

Step 1: Add the Google Fonts stylesheet to your page's <head>:

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Nunito:wght@200..1000&display=swap" rel="stylesheet">

In WordPress, paste these into your theme's header (Appearance > Theme Editor > header.php, or use a plugin like Insert Headers and Footers).

Step 2: Reference the font in your theme's --hrv-font-family variable:

{
  "name": "My Theme",
  "harvest_version": 1,
  "variables": {
    "--hrv-font-family": "'Nunito', system-ui, sans-serif"
  }
}
Bundled vs. Google Fonts

Bundled fonts are self-contained - these themes will work offline and on any site without extra setup. Google Fonts are lighter to distribute (no woff2 in the zip) but require an internet connection and a page-level <link> tag on every site that uses the theme. Bundled fonts are recommended for shared themes; Google Fonts work well for personal use on sites you control.

Dark mode

Cards automatically follow prefers-color-scheme with no configuration. The Default theme includes appropriate dark overrides built in.

In a custom theme JSON, dark_variables specifies only the values that differ from variables - everything else falls through to the light-mode value:

{
  "name": "My Theme",
  "harvest_version": 1,
  "variables": {
    "--hrv-color-surface": "#f5f5f5",
    "--hrv-color-text": "#1a1a1a",
    "--hrv-color-primary": "#0066cc"
  },
  "dark_variables": {
    "--hrv-color-surface": "#1a1a2e",
    "--hrv-color-text": "#e0e0ff"
  }
}

If dark_variables is omitted entirely, the same values are used in both modes.

Creating custom themes

The panel's Themes tab is where you create, edit, preview, import, and export custom themes.

To create a new theme, click New theme in the Themes tab. You can start from scratch or duplicate an existing theme as a starting point. The Default theme is the best base for most work.

Edit the raw JSON in the code panel. The live preview updates immediately. Use the renderer type dropdown and Light/Dark/Auto toggle to check different card types in both color modes before saving.

A few practical notes:

  • Only include variables you're actually changing. Missing variables fall back to base-card defaults.
  • "--hrv-font-family": "inherit" makes cards match whatever font your host page uses.
  • Invalid JSON is rejected on save - the panel will show where the syntax error is.
  • Themes can be assigned to multiple tokens. Changing the theme updates all of them live.

Sharing themes

Click Export .zip on any theme to download it as a .zip file containing theme.json and, if applicable, renderer.js. Anyone can import it on their own HArvest instance via the Themes tab Import .zip button. This is how third-party themes are distributed.

User-contributed themes

Beyond the five bundled themes, HArvest ships a growing collection of extra community themes in a User Contributed Themes directory. These are not installed automatically - you import the ones you want through the panel.

Where to find the directory:

  • Inside the release.zip attached to each GitHub release, in the User Contributed Themes/ folder.
  • Or browse and download it directly from the repository on GitHub.

Each theme is provided as a ready-to-import .zip (for example neu.zip or material-eww.zip) alongside an unpacked folder of the same name. To install one, open the panel's Themes tab, click Import .zip, and choose the theme's .zip file. It then appears in the theme picker like any other theme and can be assigned to tokens. Themes that include a renderer.js also bring their own card layouts (see distributing a theme with a renderer).

Themes currently included: (more themes to be added regularly)

Keeping a contributed theme up to date

These themes live in the repository rather than being delivered through HACS, so they do not auto-update with the integration. To pick up a newer version, download the latest .zip and import it again - the panel will offer to replace your existing copy.

Custom renderers

A custom renderer replaces the built-in card UI for a specific entity domain or device class. You get full control over the HTML structure, styles, and behavior inside the Shadow DOM.

BaseCard API

All renderers extend HArvest.renderers.BaseCard. Implement three methods:

MethodWhen calledDescription
render()Once, on card creationBuild the shadow DOM using this.root and this.def. Set up event listeners. Called when the entity definition arrives.
applyState(state, attributes)Every state changeUpdate the UI to reflect the new entity state. Should be fast - avoid full re-renders.
predictState(action, data)When user triggers an actionOptional. Return an optimistic state object for instant UI feedback before the server confirms the action. Return null to skip.

The entity definition is available as this.def inside render():

FieldTypeDescription
entity_idstringFull HA entity ID
friendly_namestringHuman-readable name
domainstringEntity domain (light, fan, etc.)
device_classstring | nullHA device class if set
capabilities"badge" | "read" | "read-write"Whether the token allows control. Badge entities are routed to the badge renderer, not the domain renderer.
supported_featuresstring[]Feature flags (e.g. ["brightness", "color_temp"])
feature_configobjectFeature-specific config (e.g. min/max color temp)
gesture_configobjectTap/hold/double-tap action config
companionsstring[]List of companion entity refs
graph"line" | "bar" | "step" | nullHistory graph type. Step is available for binary sensors.
hoursnumberHistory graph time window
animatebooleanAnimate fan icon

Extending a built-in renderer

class MyLightCard extends HArvest.renderers.LightCard {
  render() {
    super.render();
    const note = document.createElement("p");
    note.textContent = "Custom footer text";
    note.style.cssText = "font-size: 11px; opacity: 0.6; margin-top: 8px;";
    this.root.querySelector("[part=card-body]").appendChild(note);
  }
}

HArvest.registerRenderer("light", MyLightCard);

Writing a renderer from scratch

class MinimalLightCard extends HArvest.renderers.BaseCard {
  #toggleBtn = null;

  render() {
    this.root.innerHTML = `
      <style>
        :host {
          display: block;
          padding: var(--hrv-card-padding, 16px);
          background: var(--hrv-color-surface);
          border-radius: var(--hrv-card-radius);
          font-family: var(--hrv-font-family);
        }
        button {
          font-size: var(--hrv-font-size-m);
          padding: 8px 16px;
          background: var(--hrv-color-primary);
          color: var(--hrv-color-on-primary);
          border: none;
          border-radius: var(--hrv-radius-s);
          cursor: pointer;
          width: 100%;
        }
        .name {
          font-size: var(--hrv-font-size-s);
          color: var(--hrv-color-text-secondary);
          margin-bottom: 8px;
        }
      </style>
      <div part="card">
        <div part="card-name" class="name">${HArvest.esc(this.def.friendly_name)}</div>
        <button part="toggle-button" type="button">Toggle</button>
      </div>
    `;

    this.#toggleBtn = this.root.querySelector("[part=toggle-button]");
    this.#toggleBtn.addEventListener("click", () => {
      this.config.card?.sendCommand("toggle");
    });
  }

  applyState(state, attributes) {
    if (this.#toggleBtn) {
      this.#toggleBtn.textContent = state === "on" ? "Turn off" : "Turn on";
    }
  }

  predictState(action, data) {
    // Return optimistic state so UI updates instantly before server confirms
    if (action === "toggle") {
      return { state: this.lastState === "on" ? "off" : "on" };
    }
    return null;
  }
}

HArvest.registerRenderer("light", MinimalLightCard);

Security note

Custom renderers are display-only. They cannot bypass server-side entity permission enforcement. A custom renderer for alarm_control_panel won't receive data for that entity - the integration blocks Tier 3 domains at the protocol level regardless of what the renderer requests.

Renderer overrides

A renderer override is a renderer.js file bundled inside a theme zip. It registers multiple custom renderers at once. Renderer overrides apply only to the theme they ship with - activating a different theme deactivates the override.

The key difference from calling HArvest.registerRenderer() directly in a page:

  • Renderer overrides apply only to tokens assigned the theme that includes the override, not to all cards on the page.
  • The server activates the override automatically when the widget authenticates with a token using that theme.
  • No page-level HTML change is needed by the site owner.

Renderer registration in renderer.js

Renderers are registered into window.HArvest._renderers[rendererKey] - a separate namespace from the global renderer registry. This keeps renderer overrides scoped to tokens using that theme:

// The IIFE pattern avoids polluting the global scope
(function () {
  "use strict";

  class DialLightCard extends HArvest.renderers.BaseCard {
    render() {
      // ... custom dial-based light UI
    }
    applyState(state, attributes) {
      // ... update dial position
    }
  }

  class MinimalFanCard extends HArvest.renderers.BaseCard {
    render() {
      // ... compact fan card
    }
    applyState(state, attributes) {
      // ...
    }
  }

  // Register using your theme's ID as the renderer key
  window.HArvest = window.HArvest || {};
  window.HArvest._renderers = window.HArvest._renderers || {};
  window.HArvest._renderers["my-theme-id"] = {
    "light": DialLightCard,
    "fan":   MinimalFanCard,
    "sensor.temperature": MyTempCard,
  };
})();

Renderer lookup order

When a card needs to render, it checks in this order:

  1. The active theme's renderer for this domain + device class (e.g. "sensor.temperature")
  2. The active theme's renderer for this domain only (e.g. "sensor")
  3. The global registry (renderers registered via HArvest.registerRenderer())
  4. Built-in renderer for the domain
  5. Generic Tier 2 fallback

Re-render on activation

When a renderer override is activated, the widget automatically re-renders all active cards. Cached state and history data are replayed so cards populate immediately without waiting for the next state update.

Pairing a renderer with a theme

Bundle your theme definition and renderer into a .zip file and import it via the Themes tab. The files inside the zip must be named exactly theme.json and renderer.js - these are fixed filenames, not placeholders. The zip file itself can be named anything (e.g. dial-theme.zip).

{
  "name": "Dial Theme",
  "harvest_version": 1,
  "description": "Dial-based light cards and compact fan controls",
  "variables": {
    "--hrv-color-primary": "#00bcd4",
    "--hrv-color-surface": "#1a1a2e",
    "--hrv-color-text": "#e0e0ff"
  }
}

The has_renderer flag is set automatically on import when renderer.js is present in the zip. You do not need to set it manually.

Distributing a theme with a renderer

A theme with a renderer override is distributed as a single .zip file. The zip always contains theme.json (required) and renderer.js (the custom renderer). It may also include .woff2 font files and a thumbnail.png or thumbnail.jpg (preview image). Name the zip file itself after your theme (e.g. dial-theme.zip).

  • Share the .zip directly (GitHub release, repo attachment, etc.)
  • Users import it via the HArvest panel Themes tab - Import .zip
