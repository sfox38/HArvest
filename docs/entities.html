Entity types

Covers which entity domains are supported, what the dedicated card renderers do, what commands are available, and which domains are blocked for security reasons.

Tier overview

HArvest categorizes every HA entity domain into one of three tiers. The tier determines what UI is shown and whether the domain can be used at all.

TierDescription
Tier 1 Dedicated card renderer with full UI controls. State displays, interactive controls, and history graph support where applicable.
Tier 2 Generic display-only card - shows icon, entity name, and current state text. Any domain not in Tier 1 or 3 lands here.
Tier 3 Blocked. Cannot be added to any token. The integration rejects these at token creation time.

Sensitive domains

A subset of Tier 1 domains are disabled by default because their actions carry higher real-world risk. An admin must explicitly enable each one in Settings > Security > Sensitive domains before entities from that domain can be added to any token.

DomainWhy it's gated
lockControls physical locks - enabling this can open doors
coverControls garage doors, gates, and motorized blinds
scriptExecutes arbitrary HA script logic
automationTriggers, enables, or disables HA automations
button, input_buttonOne-shot press actions, often wired to scripts or scenes. Both domains are controlled by the Button sensitive-domain setting.

When a domain is disabled, entities from it are hidden from the entity picker, rejected if included in a token creation or update request, and silently dropped from any active WebSocket session that already holds them. Each domain can be enabled independently - you can allow lock without also allowing script.

Tier 1 - Dedicated renderers

Every entity has universal settings available in the entity settings card: name override (custom display name replacing the HA friendly name), force color scheme (Auto, Light, or Dark per entity), access (Badge, Read only, or Control), companions (not available for Badge), gestures (not available for Badge), and excluded attributes. When access is set to Badge, the entity renders as a compact pill-shaped indicator showing only icon, name, and state. Badge-specific settings (show icon, show name, show state, icon color) replace the domain-specific and companion settings. Domain-specific settings are listed under each entity type below.

light

On/off toggle, brightness slider, color temperature slider. Hides controls that the physical device doesn't support (e.g. no brightness slider on a non-dimmable light). Supported features are read from the entity's supported_features bitmask.

Display settings: brightness, color temperature, and RGB color can each be individually toggled on or off in the entity settings card.

History graph: supported (configurable per entity in the panel)

switch

On/off toggle. Simple state display with device-class icon.

binary_sensor

Read-only. Displays current state with device-class appropriate icon and label (motion, door, window, smoke, etc.).

History graph: supported

sensor

Read-only. Three sub-renderers based on device class:

  • temperature - large temperature value display with unit
  • humidity - large humidity percentage display
  • battery - percentage bar with color (green/amber/red by level)
  • generic sensor - value + unit, works for any numeric sensor

History graph: supported (line or bar, configurable per entity)

fan

Toggle, speed percentage slider, oscillate toggle, direction selector (forward/reverse). The fan icon optionally spins when the entity is on (enabled via the Animate display setting in the panel). Controls that the physical device doesn't support are hidden.

Display settings: choose a display mode to override auto-detection: Auto (default), On/Off only (hides speed controls), Continuous (forces slider), Stepped (forces step buttons), or Cycle (forces cycle button).

climate

HVAC mode selector (heat, cool, heat_cool, auto, dry, fan_only, off), target temperature controls (up/down buttons with numeric display), current temperature read-only display, fan mode dropdown, preset mode dropdown. All controls respect the entity's supported modes.

Display settings: HVAC modes, presets, fan mode, and swing mode can each be individually toggled on or off.

cover

Open, stop, and close buttons. Position and tilt sliders (0-100%) are shown when the cover supports the corresponding feature. Controls are shown or hidden based on device capabilities and display settings.

Display settings: position slider and tilt control can be individually toggled.

media_player

Play/pause, next track, previous track, volume slider, mute toggle. Current track title, artist, and source display. Album art is not supported in v1.

Display settings: transport controls, volume, and source selector can be individually toggled.

input_boolean

On/off toggle. Functionally equivalent to switch but for HA input helpers.

input_number

Range slider spanning the entity's min/max, with current value display. Step size respects the entity's configured step.

Display settings: choose between Slider (default) or Buttons mode.

History graph: supported

input_select

Dropdown selector showing all available options. Current option is pre-selected.

timer

Countdown display. Start/resume, pause, cancel, and finish buttons. State-aware: shows relevant buttons for current state (idle, active, paused).

remote

Explicit on/off power control. When the remote exposes an activity_list, the card also shows an activity dropdown and highlights current_activity. Selecting an activity calls remote.turn_on with that activity. Arbitrary remote.send_command actions are available through configured gestures and custom renderers, not through a raw command field in the built-in card.

weather

Read-only. Displays current conditions by default: a large condition icon paired with the temperature, condition label, and a detail row showing humidity, wind speed, and pressure.

Forecasts are optional and disabled by default. Enable Show forecast in the weather entity's display settings to fetch and display forecast data. When enabled, HArvest fetches available daily and hourly forecasts directly from HA and subscribes to later forecast updates. The card shows a 5-day strip for daily forecasts and lets visitors switch between daily and hourly views when both are available.

This forecast integration works with HA 2024.4+ where forecast data is no longer provided as a weather entity state attribute.

Display settings: Show forecast enables or disables forecast fetching, subscriptions, and display for this entity.

History graph: not supported.

lock

Lock and Unlock buttons. Both buttons are disabled while the lock is transitioning (locking, unlocking) or jammed. The current state is shown as a text label. Read-only capability hides the buttons and shows the state label only.

States: locked, unlocked, locking, unlocking, jammed. The jammed state is highlighted in error colour.

Optimistic UI: the card predicts the transitioning state immediately after a command, then reverts to the server-confirmed state when the real update arrives.

person

Read-only. Displays the person's presence state: Home, Away, or the name of the zone they are in.

button and input_button

Both domains use the same dedicated renderer: a single full-width Press button. No ongoing state is tracked. The button is disabled when the entity is unavailable.

number

Range slider spanning the entity's min/max, with current value display. Functionally identical to input_number. Used by integrations such as smart thermostats and LED controllers.

History graph: supported

select

Dropdown selector showing all available options. Functionally identical to input_select. Used by integrations such as AV receivers and tuners.

script

Run button that triggers the script. If the script defines input variables (via fields: in the YAML config), the wizard and token editor show a typed form for each variable - number inputs, dropdowns, entity pickers, area selectors, and more, matching the field's HA selector type. Default values are pre-filled. These values are baked into the token at creation time; visitors cannot change them. The card shows a pulse animation while the script is running.

States: off (idle), on (running).

automation

Trigger button that fires the automation regardless of its on/off state. An enable/disable toggle lets visitors turn the automation on or off.

States: on (enabled), off (disabled).

Tier 2 - Generic cards

Any domain not in Tier 1 or Tier 3 gets the generic card renderer. It shows:

  • Domain icon (MDI, best-match for the domain)
  • Entity friendly name
  • Current state as text

Some examples of Tier 2 domains: text, date, time, event, calendar.

Tier 2 domains remain display-only in the generic card. Admins can register allowed services under Settings > Custom Domains so configured gestures or a custom renderer can send commands, but this does not add controls to the generic card. See the Custom domains section for details.

Tier 3 - Blocked domains

These domains cannot be added to any token. The integration rejects them at token creation time in the panel - they won't appear in the entity picker at all.

DomainWhy blocked
alarm_control_panelSecurity system - only arm/disarm from trusted interfaces
device_trackerPrivacy concern - exposes real-time location. Use person domain instead
cameraLive video feed - significant bandwidth and privacy risk
sceneDeferred to v2 review
updateHA update management - admin-only operation
Need to trigger something in a blocked domain?

Use an HA script entity. Create a script in Home Assistant that wraps the service calls you need, then add that script entity to a HArvest token. The visitor gets a trigger button; the script controls what happens server-side.

Companion-eligible domains

Only these domains can be set as companions (secondary entities displayed inside another card):

DomainAccess in companion roleTypical use
lightToggle on/offAdditional light in the same room shown as a pill
switchToggle on/offRelated switch shown alongside a primary entity
input_booleanToggle on/offBoolean helper state
fanToggle on/offFan power state alongside a climate or light card
binary_sensorRead-onlyMotion, door/window state, occupancy indicator inside a light card
sensorRead-onlyTemperature next to a climate card, power draw next to a switch
lockLock/UnlockLock state shown alongside a door sensor or light card
coverRead-onlyCover position indicator alongside a related entity
remoteRead-only power stateRemote power status alongside a media player card

There is no per-card companion limit. The total number of entities (primaries plus companions) per token is capped at 50. Companions are configured per-entity in the panel's entity display settings - no HTML attribute needed. When the companion is assigned, the server automatically subscribes it and delivers its state alongside the primary entity.

Important

Each companion has a "Read only" toggle that defaults to on. When read-only, the companion shows state but cannot be interacted with, regardless of the domain. If the primary card's access is set to "Read only", all its companions are forced to read-only and the toggle is hidden.

Custom domains

The built-in allowed services table covers Tier 1 domains only. For a Tier 2 domain, an admin can register specific allowed services in the HArvest panel under Settings > Custom Domains. This is useful when a configured gesture or custom renderer needs to control a domain such as vacuum, humidifier, or valve.

Custom domain registration is a global server setting. It permits the selected services for all HArvest widgets, but only when the target entity is included in the widget's token with Control access. It does not add entities to tokens, change Read only entities to Control, or add controls to the generic card.

To register a custom domain:

  1. Open the HArvest panel and go to Settings > Custom Domains.
  2. Select a domain from the dropdown. It lists Tier 2 domains that currently have at least one entity and registered HA service. Tier 1 domains already have built-in service lists, and Tier 3 domains cannot be added.
  3. Select only the HA services that the gesture or renderer needs.
  4. Click Add. The allowlist takes effect immediately, including for active widget sessions.

To change the allowed services later, remove the domain and add it again with the new selection. Removing a domain immediately rejects later commands for it, but does not disconnect active widget sessions.

Custom Domains changesCustom Domains does not change
Which services the HArvest server will accept for a Tier 2 domainWhich entities are included in a token
Whether configured gestures and custom renderers can send those servicesAn entity's Badge, Read only, or Control access
The allowlist used by all current and future widget sessionsThe display-only behavior of the built-in generic card
Important

A permitted custom-domain command must pass every HArvest check: the entity must belong to the token, its access must be Control, the service must be selected in Custom Domains, and the command must remain within the token's rate limits. HArvest filters service data against the service schema registered in HA and silently removes unknown data keys.

Tip

Only enable services your widget actually uses. Each allowed service is a potential action a visitor with a qualifying widget can trigger. Display-only Tier 2 cards do not need Custom Domains registration.
